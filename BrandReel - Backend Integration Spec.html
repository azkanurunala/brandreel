Backend Integration Spec

How to turn the BrandReel prototype's mocked publishing, OAuth, and analytics into production API calls — the services to build, the secret boundary, the per-platform contracts, and the exact REST surface the client already expects.

Read this first

The prototype is a client-side React app. Two integrations in it are already real — Claude (hook scripts + caption adaptation) and Google Veo (clip generation). Everything else — social publishing, OAuth sign-in, live insights — is a faithful mock. None of those can run from the browser: they require server-held secrets and platform-approved apps. This doc specifies the BrandReel API backend that brokers them.

The client never holds a platform token or an AI key. It authenticates to your API with a first-party session; the backend holds all third-party credentials, performs token exchange/refresh, enforces rate limits, and owns the publish + analytics pipelines.

Backend components to build

  • Auth service — first-party sessions + OAuth broker for each platform (start, callback, refresh, revoke).
  • Secret store — encrypted token vault (KMS-backed); never log tokens.
  • Render workers — call Veo, transcode to each platform's spec, store to object storage + CDN.
  • Publish workers + scheduler — per-platform adapters, stagger/rate-limit queue, retry with backoff.
  • Webhook sink — receive async publish/transcode callbacks from platforms.
  • Insights jobs — scheduled pulls of post metrics; aggregate to the Insights screen's shape.

§3 · The secret boundary

Each platform's publish scope requires an OAuth 2.0 authorization-code flow whose code-for-token exchange uses a client secret. That exchange must happen on the server. The browser only ever sees a short-lived redirect and your first-party session cookie.

Generic connect flow

1 · StartClient hits POST /auth/:platform/start; API returns the provider consent URL (with state, PKCE).
→
2 · ConsentUser approves on the platform. Maps to the prototype's OAuth consent sheet.
→
3 · CallbackProvider redirects to /auth/:platform/callback; server swaps code→tokens, stores them, links the connection.
# server-side token exchange — secret never leaves the backend
POST https://open.tiktokapis.com/v2/oauth/token/
  client_key=$TIKTOK_CLIENT_KEY  client_secret=$TIKTOK_CLIENT_SECRET
  grant_type=authorization_code  code=<code>  redirect_uri=<cb>  code_verifier=<pkce>
# → { access_token, refresh_token, expires_in, open_id, scope }  → vault
Token refresh = the prototype's "token refreshed 7d ahead"

Run a scheduler that refreshes each connection before expiry and flips the connection's status to expiring / expired when a refresh fails — that's exactly the state the Profile & Inbox screens render today.

§5 · Publishing adapters

One adapter per platform behind a common interface. Each enforces the format contract already encoded in the prototype (ratio, maxSec, capMax, hashtags) and returns a normalized result the publish pipeline can track.

interface PublishAdapter {
  validate(asset, caption): Preflight       // aspect · duration · codec · caption length
  publish(connection, asset, caption): { remoteId, permalink, state }
  fetchInsights(connection, remoteId): { views, likes, comments, shares, reach }
}

TikTok · 9:16 · ≤60s

PUBLISH

Content Posting API: /v2/post/publish/video/init/ → chunked upload → poll /publish/status/fetch/. Caption ≤150 chars, 5 tags.

Instagram · 9:16 · ≤90s

PUBLISH (2-step)

Create container POST /{ig-user}/media (media_type=REELS, public video_url, caption) → POST /{ig-user}/media_publish. Caption ≤2200.

YouTube · 9:16 · ≤60s

UPLOAD

videos.insert resumable upload; title/description, #Shorts. Title-line ≤100 chars in the prototype's caption slot.

LinkedIn · 1:1 · ≤90s

REGISTER → UPLOAD → POST

Register video asset → upload binary → create a Post referencing the asset. Copy ≤700, 3 tags.

X · 16:9 · ≤140s

MEDIA UPLOAD → TWEET

Chunked media upload (INIT/APPEND/FINALIZE) → POST /2/tweets with media_ids. Copy ≤280, 2 tags.

Facebook · 1:1 · ≤90s

PAGE VIDEO POST

POST /{page-id}/videos with Page token + description. Caption ≤2200, 4 tags.

§6 · Publish pipeline & scheduler

The "Publishing" screen's queue → posted → retry → failed transitions become real worker states:

  1. Pre-flight — run each adapter's validate(). This is the prototype's pre-flight checklist (aspect/duration/codec, token validity, rate-limit clearance, dedupe, voice-match).
  2. Stagger / rate-limit — the scheduler spaces posts (e.g. TikTok 1 / 30 min) to avoid shadow-bans — the "staggered to avoid rate limit" copy is literally this queue.
  3. Publish — adapter posts; persist remoteId + permalink.
  4. Retry — on 429/transient errors, exponential backoff (the "X · 429 rate-limited, retry in 60s" log row).
  5. Auto-fix — on a spec rejection (the "IG aspect 9:16 expected, got 9:15" case), re-transcode/smart-crop and re-queue rather than hard-failing.
Map directly to existing UI states

Publish status enum queued · posted · retry · failed already drives br-screens-campaign.jsx. Emit these verbatim and the Publishing screen lights up with zero UI changes.

§9 · Insights ingestion

Replace the "Momentum" mock with a scheduled job per live Post: pull metrics from each platform's analytics endpoint (or receive via webhook), write an InsightSnapshot, and aggregate into the Insights screen's existing shape — totals, by-platform, by-hook-angle, top performer.

  • TikTok video query / insights · Instagram media insights · YouTube Analytics API · LinkedIn social actions · X tweet metrics.
  • Cadence: dense (5–15 min) for the first 2 hours — the "monitoring first 2 hours" copy — then taper to hourly/daily.
  • Derive the recommendation card ("Hook 2 underperforms on LinkedIn") from by-hook×platform deltas.

§10 · AI layer — already real, keep server-side

Move both providers behind the API so keys never ship to the client; prompts are unchanged from the prototype.

  • Claude — the campaign prompt (5 hooks + captions + hashtags) and the per-platform caption adapter in br-ai.jsx. Keep the JSON-shape contract + template fallback.
  • Veo — model veo3.1-fast, aspect from platform (9:16 / 1:1 / 16:9), 4–8s. Persist outputs and register them in the same manifest shape (campaign→hook→ratio→url) the client already reads (br-veo-manifest.jsx). Canvas render stays as the instant fallback.

§12 · Suggested rollout

  1. Phase 0 — stand up the API + session auth; proxy Claude & Veo (client behavior unchanged, keys moved server-side).
  2. Phase 1 — OAuth + Connections for one platform end-to-end (TikTok). Start app review for TikTok + Instagram in parallel.
  3. Phase 2 — publish pipeline + scheduler for the approved platform; wire the Publishing screen to real /status.
  4. Phase 3 — remaining platforms as reviews clear; insights ingestion → real Insights screen.

Net: the client is production-shaped already — its enums, screens, and the Veo manifest are the integration contract. The work is a backend that speaks them. Build the API, keep the front end; swap mocks for endpoints one phase at a time.
